Pascal-S 语言编译程序的设计与实现 
1. 课程设计任务和目标  
按照源语言 Pascal-S 的语法和语义，设计并实现 Pascal-S 到 C 语言的编译程序。
要求生成的 C 语言程序与 Pascal-S 语言程序面对相同输入的输出结果一致，同时对错
误 Pascal-S 程序，可以进行错误处理与错误恢复，提供尽可能详细丰富的报错信息。 
2. 需求分析 
2.1 开发环境 
a. Flex  2.6.4 
b. Bison 3.8.2 
c. C++ 14  
d. Python 3.8.16 
2.2 功能分析 
• 实现 Pascal-S 语言到 C 语言的转换 
• 支持 Pascal-S 语言的常见数据类型，包括整数、实数、布尔值和字符，扩展实现
了字符串等数据类型 
• 支持 Pascal-S 语言常见流程控制结构，包括分支语句、循环语句、函数和过程等 
• 支持 Pascal-S 语言的标准库函数和自定义函数 
• 实现 Pascal-S 语言的编译器前端，包括词法分析、语法分析和语义分析 
 
2.3 非功能分析 
• 性能要求高，能够快速地将 Pascal-S 代码转换为 C 代码 
• 代码质量要高，生成的 C 代码应该准确无误、结构清晰 
• 可靠性高，能够处理各种 Pascal-S 语言的代码和错误情况 
• 进行详细的测试和验证，确保转换后的 C 代码和原始 Pascal-S 代码具有相同的运
行结果

￮ 错误处理模块

3.1.2 模块功能 
a. 词法分析模块 
      从输入文件中读取字符流。并根据 Pascal-S 语言的词法规则，将字符流分解
成单词（token）序列。 
b. 语法与语义分析模块 
根据 Pascal-S 语言的语法规则，将 token 序列按语法规则进行识别，并生成
符和抽象语法树（AST） ，并进行类型检查和错误恢复。 
c. 代码生成模块 
该模块的主要功能是将语法与语义分析模块输出的语法树转换成等效的 C 语
言代码。 
d. 错误处理模块 
该模块的主要功能是在编译过程中检测并处理各种错误，如词法、语法、语
义等方面的错误。对于检测到的错误，该模块需要记录其位置、类型，并输出相
应的错误信息。 
e. 测试模块 
该模块的主要功能是对编译程序的各个模块进行测试，包括单元测试和集成
测试。

4.1.2 功能描述 
从输入的 Pascal-S 源程序中读取字符流，并根据该语言的词法规则，将字符流分割成
一系列词法单元并输出记号流，其中每个记号包含类型、值、长度、行号以及列号。
具体功能如下： 
识别 Token

• true、false：m_BOOLEAN

4.2 语法语义分析 
4.2.1 接口描述 
Plain Text 
yyparse (pascals::ast::AST *real_ast) 
语法语义分析程序通过 yyparse(pascals::ast::AST *real_ast)函数调用。对于词法分析
的衔接，是通过不断地调用 yylex()函数来获取 Token流进行分析，在分析的过程会生
成符号表，语法树等结构，为后续代码生成保留必要的信息。 
而 yyparse()的返回值取决于是否含有语法错误，如果没有语法错误，返回值为 0，这
时我们将调用代码生成模块。 
如果产生语法错误，返回值为非零，将不再调用代码生成，结束编译程序.同时我们也
会对 ast 的根节点进行设置，如果语法语义分析找到错误，那么程序会将 ast 的根节点
设为空指针，这时我们也不会进行代码生成. 
4.2.2 功能描述 
在此阶段，程序主要完成的任务是将从词法分析读到的 Token 流解析为文法表达
式，并通过相应的翻译方案和语义动作，完成符号表的插入与查询、类型检查、错误
处理与恢复，并构建一棵语法树，提供给代码生成模块。 
我们认为课本上的文法与 PPT 上的文法皆有不全之处，因此我们在二者的基础上，
增添了许多实用的文法，可以帮助用户更见便捷地编写代码。我们文法产生式共122
条，详情请参见附录。

             std::is_same<T, BasicType>::value || 
             std::is_same<T, RecordType>::value) { 
    auto type_entry = def_types_.Find<T>(name); 
    if (type_entry != nullptr) return (T*)type_entry; 
  } 
 
  if (local_zone != nullptr) *local_zone = false; 
  if (prev_table_set_ != nullptr) 
    return prev_table_set_->SearchEntry<T>(name); 
  return nullptr; 
}

复杂类型的访问 
在课本提供的文法中，我们通过与 id_varparts 相关的产生式实现复杂类型的访问。
相关的产生式既可以推出数组的按下标访问（如 a[i]） ，也可以推出结构体成员访问
（如 a.i） ，这两者展开时完全可以重复地混在一起（a[i].b.c[j]...） 
数组下标访问的检查与结构体成员访问要求检查的逻辑是明显存在差别的，各自
要检查的要点各自独立。比如，经过研究，小组发现数组每一维的下标在声明时便被
确定，合法访问需要类型检查（比如对于 a['b']这种 char 类型的下标，a[1]式的访问是
非法的） ；结构体成员的具体命名自然是任意的，实际检查中一个朴素的需求便是要检
查该成员是否存在，这需要查询符号表。 
进一步地，我们研究发现，语法在多维数组访问时，实际上既支持 a[1][2]式的访
问，又支持 a[1,2]式的访问，这意味着单次'[]'内的访问可以是非法的，但展开仍然要继
续，不确定是否还会再读到下一个'[]'转为合法情况。 
诸多的这些特别要素混在一块，使得对复杂类型的访问方式有待仔细考量。本小
组的实现方式是 S 属性式的：在展开的过程中，记录所有的展开形式（是数组还是结
构体？）和需要记录的展开时的内容（下标[]访问中读到的是什么下标类型、有几个？
数组访问中访问的是哪个成员？） 。直到归约为id id_varparts 这个层面时，才对这一
次具体的复杂类型访问进行检查，此时类型检查的所有内容都被收入 IDVarpartsAttr 内
含的 VarParts 容器中，可以通过 IDVarpartsAttr::AccessCheck函数实现类型检查。

4.4 错误处理 
4.4.1 概述 
错误处理模块采取错误恢复策略：当程序检测到错误时不会停止编译，而是跳过
若干个符号之后继续进行编译。同时，在编译过程中，程序将记录识别到的每一种错
误的错误码、出现在源程序中的行号以及列号，并直接进行输出。之所以可以直接输
出做是因为 yyparse()与 yylex()之间的调用关系。词法分析和语法语义分析的共同入口
是 yyparse()，即语法分析的主程序，这个程序不需要收到一个完整的记号流，只在需
要记号的时候调用 yylex()函数返回一个记号。这样做保证了我们错误一定是按行按列
有序输出的，我们就不需要将全部错误进行存储再进行排序输出。 
当前错误类型主要基于Pascal-S APPENDIX C: Explanations to Error 
Codes，详见 6.3 测试情况。

bison 的错误处理机制： 
先给出结论： 
当当前栈顶状态无法移进向前看符号时，程序会先尽可能地采取默认规约动作，
将栈顶状态进行规约以希望得到一个能够规约向前看符号的状态；直到规约到某个状
态，它既不能规约，也不能移进。此时，分析程序会压入一个 error 符号到输入符号串
之前，即当前的向前看符号变为了 error；这时，分析程序会判断栈顶状态是否可以移
进 error，如果如果不可以，就退栈，重复此操作，直到找到一个状态可以规约掉error
符号。 
在没有设计 error 产生式时，一旦出现了 error，bison 分析程序不可能从栈中找到
一个状态去移进 error 符号，这时会退栈到空栈状态，然后抛弃掉所有后续符号。 
因此，我们的主要工作就是设计合理的 error 产生式去匹配我们希望进行恢复的错
误。

如何设计 error 产生式 
在实际使用 error 产生式时，我们发现，如果过于精细地用 error 产生式去匹配出错的
情况，非常容易引入移进规约冲突和规约规约冲突： 
在经过大量的试错，调整再试错之后，我们总结了一些使用 error 产生式的经验： 
a. 自顶向下,逐步求精：即在设计错误产生式时，我们先要做到将错误定位到一
个大的语法结构中去解决，如常量定义模块，类型定义模块，变量定义模块等，
在这些模块中，我们大可采用一些比较激进的方法，比如直接抛弃整个模块。做
到这里，我们就可以保证在每个模块只出现一个错误时，我们是可以全部定位的。
接下来再在每个模块中去定位错误。 
b. error 产生式尽量不要改变 FOLLOW 集，即把 error 符号放在终结符的后面，
我们从大量的试错中总结出，大部分引进规约冲突都是由于引入的 error 符号直接
放在了非终结符的后面 如A→B error b 这使得 B 的 FOLLOW 集中有了 error,很

4.4.4 语义分析阶段 
语义阶段错误可以分为以下几类： 
• 与符号表相关的错误：如未定义、重定义 
• 类型检查错误 
关于变量的声明问题，我们对程序的要求如下： 
• 对于符号名称，我们不允许同层之间同名，但是允许与全局变量同名 
• 对于类型名称，我们既不允许同层之间同名，也不允许与全局的类型名称同名 
如果违背上述要求，则报错

7.2 主要问题与解决方案 
• 二义文法： 
因此我们通过删除一条产生式解决了规约/规约冲突。而移进/规约冲突是由于 if-
else 文法的二义性产生的，经查阅资料得知，在不做任何处理的情况下，bison 编
译器遇到移进/规约冲突会默认按照移进进行处理，这样实际上相当于帮助我们实
现了最近匹配原则，因此我们无需作任何处理。 
• 类型符号表： 
在我们的代码实现过程中，由于我们引入了 record 类型，加之 array 类型，我们
的 type 非常的纷繁复杂，所以在类型检查的时候，两个类型之间能否执行某些运
算，能否赋值就给我们的类型检查带来了很大的负担，这正是我们引入类型符号
表的原因，通过引入类型符号表，我们能够真正完成数组的结构等价以及其他类
型的名字等价，类型符号表的具体设计详见 4.2.3。 
• 标准库函数： 
在我们的需求中，我们希望能够在翻译过程中正确翻译 pascal-S 中的所有基本库
函数，包括 abs,sin,read,write 等等。为了实现这一目的，我们将所有的标准库函

7.3 改进建议 
• 文法：我们小组虽然在 PPT 与课本文法之上做了许多补充完善，但是 Pascal-S
文法仍存在不宜使用之处，如 if -then 语句后面只能用 statement，对分号的使用有一
定限制。 
• 语义分析：本小组将语义分析与语法分析融合完成，导致 yacc 文件中代码冗长，
存在部分重复函数。将语义分析转移至 yacc 生成语法树后完成，能够使得结构更清晰。 
• 更多的 model C++，例如使用 shared_ptr 代替 C++原生指针，使用 variant 代替
union，可以很好地减少手动释放内存的工作量以及避免内存泄漏的问题。
